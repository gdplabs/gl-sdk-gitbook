# Metrics Aggregator

`MetricsAggregator` is the universal aggregation engine behind every evaluator (`GEvalGenerationEvaluator`, `AgentEvaluator`, `CompositeEvaluator`, and custom evaluators extending `BaseEvaluator`). After all metrics complete, the evaluator passes their results through `MetricsAggregator` to produce two outputs:

| Output              | Type    | Logic                                                                           |
| ------------------- | ------- | ------------------------------------------------------------------------------- |
| `aggregate_success` | `bool`  | AND-gate — `True` only if **every** metric passed                               |
| `aggregate_score`   | `float` | Polarity-aware mean — inverts `higher_is_better=False` metrics before averaging |

These two values then feed into `aggregate_explanation` (generated by the evaluator, not the aggregator) — a human-readable summary listing which metrics failed and why.

```
┌─────────────────────────────────────────────┐
│  Evaluator._aggregate_metrics()              │
│                                              │
│  Metrics run → MetricResult per metric       │
│       ↓                                      │
│  MetricsAggregator.aggregate()               │
│       ↓                                      │
│  AggregationResult(                          │
│    aggregate_success: True/False,            │
│    aggregate_score: 0.0–1.0                 │
│  )                                           │
│       ↓                                      │
│  Evaluator._generate_explanation()           │
│       ↓                                      │
│  EvaluatorResult (aggregate_explanation)     │
└─────────────────────────────────────────────┘
```

{% hint style="info" %}
Every evaluator uses `MetricsAggregator` by default. You don't need to create one — evaluators instantiate it internally. You only interact with it directly when passing a custom aggregator (e.g., `WeightedMetricsAggregator`) to an evaluator, or when subclassing for custom logic.
{% endhint %}

***

### How It Works

The `MetricsAggregator` has two overridable hooks:

1. **`compute_success(named_results) → bool`** — AND-gate: all metrics must return `success=True`. Empty results → `False`.
2. **`compute_score(named_results) → float`** — polarity-aware mean. For each metric:
   * `higher_is_better=True` → uses score as-is
   * `higher_is_better=False` → inverts (`1.0 - score`)
   * Returns `0.0` if empty.

These are called by the public `aggregate()` method which wraps them into an `AggregationResult` dataclass.

#### AggregationResult

```python
@dataclass
class AggregationResult:
    aggregate_success: bool  # True if ALL metrics passed
    aggregate_score: float   # Polarity-aware mean of scores
```

***

### Basic Example

```python
from gllm_evals.aggregation.metrics_aggregator import MetricsAggregator
from gllm_evals.types import GEvalMetricResult

aggregator = MetricsAggregator()

named_results = {
    "completeness": GEvalMetricResult(
        score=1.0,
        rubric_score=3,
        success=True,
        higher_is_better=True,
    ),
    "redundancy": GEvalMetricResult(
        score=0.2,  # Lower is better for redundancy
        rubric_score=1,
        success=True,
        higher_is_better=False,
    ),
}

result = aggregator.aggregate(named_results)
print(result)
# AggregationResult(aggregate_success=True, aggregate_score=0.9)
# score = (1.0 + (1.0 - 0.2)) / 2 = 0.9
```

***

### Polarity-Aware Scoring

Built-in metrics with `higher_is_better=False` (inverted before averaging):

| Metric                          | Why inverted                          |
| ------------------------------- | ------------------------------------- |
| `GEvalRedundancyMetric`         | Lower redundancy is better            |
| `DeepEvalBiasMetric`            | Lower bias is better                  |
| `DeepEvalToxicityMetric`        | Lower toxicity is better              |
| `DeepEvalPIILeakageMetric`      | Lower PII leakage is better           |
| `DeepEvalPromptAlignmentMetric` | Lower misalignment is better          |
| `DeepEvalRoleViolationMetric`   | Lower role violations is better       |
| `DeepEvalMisuseMetric`          | Lower misuse risk is better           |
| `DeepEvalNonAdviceMetric`       | Lower non-advice violations is better |

For metrics where `higher_is_better=False`, the score is inverted before averaging. A low "bad" score (e.g., 0.1 redundancy) contributes positively to `aggregate_score`.

```python
named_results = {
    "completeness": GEvalMetricResult(
        score=1.0, success=True, higher_is_better=True,
    ),
    "redundancy": GEvalMetricResult(
        score=0.3, success=True, higher_is_better=False,
    ),
}

result = aggregator.aggregate(named_results)
# aggregate_score = (1.0 + (1.0 - 0.3)) / 2 = 0.85
```

***

### AND-Gate Success Logic

`aggregate_success` is the strict quality gate: one failure fails the whole evaluation.

```python
named_results = {
    "completeness": GEvalMetricResult(
        score=1.0, success=True, higher_is_better=True,
    ),
    "redundancy": GEvalMetricResult(
        score=0.3, success=False, higher_is_better=False,
    ),
}

result = aggregator.aggregate(named_results)
# aggregate_success = False  ← redundancy failed
# aggregate_score = 0.85     ← score is still high
```

Even when `aggregate_score` is high, `aggregate_success` can be `False` — a single failing metric gates the entire evaluation. This mirrors production quality gates where all dimensions must pass.

#### Empty Results

No metrics evaluated → no confidence → fail:

```python
result = aggregator.aggregate({})
# AggregationResult(aggregate_success=False, aggregate_score=0.0)
```

### Passing Custom Aggregators to Evaluators

All evaluators accept a `metrics_aggregator` parameter in their constructor. Pass a custom aggregator to change how scores or success are computed:

```python
from gllm_evals.evaluator.composite_evaluator import CompositeEvaluator
from gllm_evals.aggregation.metrics_aggregator import WeightedMetricsAggregator

aggregator = WeightedMetricsAggregator(
    weights={"completeness": 0.7, "redundancy": 0.3},
    score_mapping={
        "completeness": {1: 0.0, 2: 0.5, 3: 1.0},
        "redundancy": {1: 1.0, 2: 0.5, 3: 0.0},
    },
)

evaluator = CompositeEvaluator(
    metrics=[completeness_metric, redundancy_metric],
    name="weighted_eval",
    metrics_aggregator=aggregator,
)
```

Same pattern works with any evaluator — `GEvalGenerationEvaluator(metrics_aggregator=...)`, `AgentEvaluator(metrics_aggregator=...)`, etc.

### Advanced: WeightedMetricsAggregator

`WeightedMetricsAggregator` extends `MetricsAggregator` to apply per-metric weights and score mappings. It overrides `compute_score` only — `compute_success` (AND-gate) is inherited unchanged.

**Key behaviors:**

* Uses `result.rubric_score` (integer, pre-threshold) instead of `result.score` (normalized float)
* Score mappings can be dictionaries (`{1: 0.0, 2: 0.5, 3: 1.0}`) or callables
* Weights must cover every metric name in `named_results`

```python
from gllm_evals.aggregation.metrics_aggregator import WeightedMetricsAggregator
from gllm_evals.types import GEvalMetricResult

weights = {
    "completeness": 0.6,
    "redundancy": 0.4,
}

score_mapping = {
    "completeness": {1: 0.0, 2: 0.5, 3: 1.0},
    "redundancy": {1: 1.0, 2: 0.5, 3: 0.0},  # Inverted: low rubric = high score
}

aggregator = WeightedMetricsAggregator(
    weights=weights,
    score_mapping=score_mapping,
)

named_results = {
    "completeness": GEvalMetricResult(
        score=1.0, rubric_score=3, success=True, higher_is_better=True,
    ),
    "redundancy": GEvalMetricResult(
        score=0.2, rubric_score=1, success=True, higher_is_better=False,
    ),
}

result = aggregator.aggregate(named_results)
# aggregate_score = (0.6 × 1.0 + 0.4 × 1.0) / 1.0 = 1.0
```

#### Callable Score Mapping

```python
def normalize_rubric(rubric_score: float) -> float:
    return rubric_score / 3.0

score_mapping = {
    "completeness": normalize_rubric,
    "redundancy": normalize_rubric,
}
```

### Custom Aggregation

Override `compute_success` or `compute_score` to implement custom logic:

```python
from gllm_evals.aggregation.metrics_aggregator import MetricsAggregator
from gllm_evals.types import GEvalMetricResult

class MinorityGateAggregator(MetricsAggregator):
    """Require at least 50% of metrics to pass (relaxed AND-gate)."""

    def compute_success(self, named_results: dict[str, GEvalMetricResult]) -> bool:
        if not named_results:
            return False
        passed = sum(1 for r in named_results.values() if r.success)
        return passed / len(named_results) >= 0.5

class MedianScoreAggregator(MetricsAggregator):
    """Use median instead of mean for aggregate_score."""

    def compute_score(self, named_results: dict[str, GEvalMetricResult]) -> float:
        scores = [
            r.score if r.higher_is_better else 1.0 - r.score
            for r in named_results.values()
        ]
        if not scores:
            return 0.0
        sorted_scores = sorted(scores)
        n = len(sorted_scores)
        if n % 2 == 1:
            return sorted_scores[n // 2]
        return (sorted_scores[n // 2 - 1] + sorted_scores[n // 2]) / 2
```

Override both for fully custom behavior:

```python
class CustomAggregator(MetricsAggregator):
    def compute_success(self, named_results):
        # Custom success logic
        ...

    def compute_score(self, named_results):
        # Custom score logic
        ...

evaluator = CustomEvaluator(
    metrics=[...],
    metrics_aggregator=CustomAggregator(),
)
```

{% hint style="warning" %}
`compute_success` and `compute_score` are the only two hooks. Do not override `aggregate()` — it wraps your overrides into an `AggregationResult` correctly.
{% endhint %}

***

### Summary

| Aggregator                    | Use Case                                                                                |
| ----------------------------- | --------------------------------------------------------------------------------------- |
| `MetricsAggregator` (default) | AND-gate success + polarity-aware mean. Used by all evaluators out of the box.          |
| `WeightedMetricsAggregator`   | Per-metric weights + rubric score mappings. Useful when some metrics matter more.       |
| Custom subclass               | Override `compute_success` (relaxed gating) and/or `compute_score` (median, max, etc.). |
